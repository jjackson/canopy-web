# Design: Canopy Web — Building Agents from Conversations

> **Terminology note:** "Building agents" is the user-facing framing — the abstract purpose of the system. Technically, the system produces **skills/playbooks** (the reusable unit) that are **managed and executed by agents** (PM supervisor, open claws, etc.). "Agent" in the title means "the thing a non-technical user thinks of when they say 'make this repeatable.'"

## Glossary

- **Open claw**: The company's autonomous AI agent runner. An open claw session is a long-running Claude Code process that operates autonomously on a task (e.g., improving a codebase, running a PM cycle). Produces JSONL session transcripts similar to Claude Code sessions. "Open claw transcript" = the session log. "Open claw runtime" = executing a skill as an open claw session.
- **Canopy**: The existing autonomous self-improving system. Lives in `~/emdash-projects/canopy/` (Python). Watches Claude Code sessions, identifies patterns, proposes improvements. This design extends Canopy.
- **Skill**: A runtime-agnostic definition of a reusable approach (the portable unit this system produces).
- **Collection**: A group of related conversation sources about the same topic.
- **Eval suite**: A living test suite attached to a skill. Grows over time.
- **Workspace session**: A collaborative human+AI session for co-authoring a skill from sources.

## Problem Statement

Knowledge about successful AI approaches dies in conversations. When someone at the company figures out a great workflow — whether through a Slack discussion, an open claw session (the company's autonomous AI agent runner, powered by Claude Code), or iterating with an AI assistant — that knowledge lives in scattered threads and transcripts. The people who could benefit most (senior leaders like the CSO, MDs, partnership heads) can't turn those approaches into reusable, runnable skills because they don't have the technical background to build Claude Code skills, MCP tools, or structured workflows.

The company has a working foundation in Canopy — an autonomous self-improving system that watches Claude Code sessions, identifies patterns, and proposes improvements. But Canopy's input surface is limited to structured AI sessions. Human conversations (Slack threads, meeting notes) and cross-source aggregation (multiple threads + sessions about the same topic) are untouched.

## Demand Evidence

**Honest assessment: This is vision-driven, not demand-validated.** No one at the company has asked "can I turn this conversation into an agent." The demand signal is observed waste — good approaches emerge and die — plus the founder's (Jonathan's) personal experience building Canopy itself by manually doing this pipeline.

**Strongest evidence:** A real Slack thread from today where Neal (CSO) shared a successful CRISPR analysis approach. Jonathan and Matt (MD) discussed how to architect it as a skill. Neal's response to "have you tried building skills on the command line?": "no and no." The knowledge exists. The translation layer doesn't.

**Risk factor:** People may not want reusable agents. They may prefer re-running conversations each time. The design must test this assumption fast.

## Status Quo

- **Jonathan (founder/builder):** Manually does the full pipeline — observes patterns in sessions, builds skills, creates evals, iterates. Canopy is the output of this process. This is the existence proof that the pipeline works, but it requires deep technical skill and significant time investment.
- **Neal, Matt, Beth (senior leaders):** See great approaches emerge in conversations. Occasionally share them on Slack. The knowledge dies there. They don't know how to build skills and haven't tried. Their workaround is to ask someone technical (like Jonathan) or ask an AI directly each time.
- **Open claw users:** Run AI sessions that succeed, but the transcript isn't analyzed or extracted into anything reusable. Each session is a one-off.

## Target User & Narrowest Wedge

**Primary target:** Neal Lesh (CSO) — senior leader who identifies valuable AI approaches but can't build skills. Represents a class of non-technical domain experts who know WHAT should be reusable but can't make it so.

**Narrowest wedge:** Neal pastes a Slack thread URL (or forwards a collection of related threads + an open claw transcript) into the web UI. The system and Neal co-author a skill together in a collaborative workspace. The skill can be run by Neal in the web UI, or by an open claw, or by anyone in Claude Code. An eval suite is extracted alongside the skill and grows as the skill is used.

## Constraints

- Must be designed generically (not company-specific), even though built for internal use first
- Multi-runtime targeting (Claude Code, open claw, cowork, web) is a day-one architectural requirement — because the target users can't use CLI tools
- Canopy is the foundation — extend, don't rebuild
- Three-surface architecture: API + web frontend + MCP layer
- The "brain" (collaborative workspace for building agents from conversations) IS the product — everything else is plumbing

## Premises

1. **Multiple sources required.** Neither conversations nor AI sessions alone are sufficient. The system must aggregate across Slack threads, AI transcripts, API logs, and other sources about the same topic. Both require iteration and eval setup.
2. **Skills managed by agents.** The reusable unit is a skill/playbook. Agents (PM supervisor, open claws, etc.) manage and execute skills. The system doesn't produce autonomous agents — it produces skills that agents run.
3. **Multi-runtime portability is day-one.** Non-technical users can't use CLI. If the extracted skill only works in Claude Code, it doesn't solve the problem. The abstraction layer that makes skills portable across runtimes IS a core product requirement. Day-one means: the skill schema is runtime-agnostic from the start, and the web + Claude Code adapters ship in V1. Open claw adapter ships alongside (similar to CC). Cowork adapter is deferred pending API spec.
4. **Evals are living test suites.** Initial eval cases are extracted from the source conversations. The eval grows as the skill is used — every successful run can become a new test case. Evals prove the skill improves over time, not just that it works once.
5. **Collaborative workspace, not pipeline.** The extraction UX is a human + AI co-authoring session, not "submit and get results." The web UI is a workbench where the work happens.
6. **"Building agents from conversations" is the single abstract purpose.** Everything in the system serves this core interaction.

## Approaches Considered

### Approach A: Workspace-First
Build the collaborative workspace UI first. Minimal backend. Prove the core interaction works before building plumbing.
- Effort: M (human: 2-3 weeks / CC: 2-3 days)
- Risk: Medium — fast feedback but no deployment
- Pros: Tests core value prop immediately
- Cons: No runtime deployment, no API, no MCP

### Approach B: API-First Platform
Build the API + MCP layer first. Define schemas, ingestion, eval framework, runtime adapters. Web UI comes last.
- Effort: L (human: 4-6 weeks / CC: 1 week)
- Risk: High — infrastructure before proof of value
- Pros: Portable from day one
- Cons: Weeks before anyone can try the product

### Approach C: Canopy Extension
Extend Canopy's existing session analysis to handle broader inputs (Slack, API logs) and add a web UI for the collaborative extraction workspace. Reuse Canopy's skill runner, proposal system, and self-improvement tracking.
- Effort: M (human: 2-3 weeks / CC: 2-3 days)
- Risk: Low — builds on proven architecture
- Pros: Massive code reuse. Canopy's patterns, skills, evals already work.
- Cons: No formal API layer. May inherit Canopy's architectural constraints.

### Approach C+B (CHOSEN): Canopy Extension + API Facade
Combines C and B: extend Canopy's engine AND expose it via a thin API facade. Web UI, MCP, and CLI are equal consumers. The API is not separate infrastructure — it's a thin layer over the Canopy engine calls.

- Effort: M-L (human: 3-5 weeks / CC: 4-5 days). Human time includes design decisions, review, manual testing, and user validation. CC time assumes Claude Code implements code under human direction. Lower than B alone because Canopy's engine handles analysis, pattern detection, and self-improvement. The new work is: conversation ingestion adapters, collaborative workspace, skill schema, runtime adapters, and the API facade.
- Risk: Low-Medium — builds on proven architecture with clear extension points
- Pros: Massive code reuse from Canopy. API is thin (not separate infrastructure). All three surfaces from day one.
- Cons: Must ensure Canopy's architecture doesn't constrain the vision.

### Core Concepts

**Source**: A conversation or transcript that contains knowledge about an approach. Types: Slack thread, Claude Code session, open claw transcript, Anthropic API log, meeting notes, Google Doc, etc.

**Collection**: A set of related Sources about the same topic. The CRISPR example = Slack thread + follow-up threads + open claw session + API logs. V1: users manually group sources into collections. Future: the system suggests related sources based on topic/keyword overlap.

**Skill**: A runtime-agnostic definition of a reusable approach. Contains: purpose, steps, inputs, outputs, tools needed, and runtime-specific adapters. The portable unit.

**Eval Suite**: A living test suite attached to a Skill. Contains: test cases (input → expected output), quality criteria, and a history of runs. Grows over time.

**Workspace Session**: A collaborative session where a human and AI co-author a Skill from a Collection of Sources. The core product interaction.

### Skill Schema (Runtime-Agnostic)

```yaml
name: crispr-analysis
version: 1.0.0
description: Analyze biological data using CRISPR-related evidence synthesis
created_from:
  - source: slack://C04ABC123/thread-1711234567
  - source: openclaw://session-abc123
  - source: gdoc://1H2810hBi9j1nfHOj36OUwgMYEHwPB6W8bxHo9MM2KmU

steps:
  - name: gather_evidence
    description: Collect relevant studies and data
    tools: [web_search, document_reader]
    inputs: [topic, constraints]
    outputs: [evidence_set]

  - name: synthesize_analysis
    description: Produce structured analysis from evidence
    tools: [llm_reasoning]
    inputs: [evidence_set, analysis_framework]
    outputs: [draft_analysis]

  - name: adversarial_review
    description: Challenge the analysis — find holes, verify claims
    tools: [llm_reasoning, web_search]
    inputs: [draft_analysis]
    outputs: [reviewed_analysis, concerns]

runtimes:
  claude_code:
    type: skill
    entry: /crispr-analysis
  open_claw:
    type: prompt_chain
    system_prompt: |
      You are a CRISPR analysis specialist...
  web:
    type: guided_workflow
    ui_steps: [upload_data, review_evidence, approve_analysis]
  cowork:
    type: task_sequence  # DEFERRED: Cowork adapter spec TBD pending cowork API documentation

eval:
  cases:
    - name: baseline_crispr_review
      input: { topic: "CRISPR gene editing in sickle cell", constraints: "peer-reviewed only" }
      expected: { contains: ["clinical trials", "safety profile"], quality: ">= 7/10" }
  growth_policy: auto
  # Growth policy "auto": When a user runs the skill and marks the result as
  # "good" (explicit approval), the input/output pair is proposed as a new eval
  # case. User confirms before it enters the suite. This ensures eval growth is
  # human-curated, not silently accumulated from every run.
```

### The Collaborative Workspace (Web UI)

The brain of the system. Not a dashboard — a workbench.

**Interaction model:** Turn-based, not real-time collaborative editing. The AI generates a complete draft; the human edits any field directly; when the human makes a significant edit, the AI re-proposes downstream changes. Think "AI writes, human edits, AI adapts" — not Google Docs-style simultaneous editing. This is deliberately scoped to avoid the complexity of real-time collaboration while preserving the collaborative feel.

**Flow:**
1. **Ingest**: User pastes source URLs (Slack thread, Google Doc, transcript file) or uploads files. For Slack: system uses Slack API (requires workspace OAuth) to fetch thread content. For Google Docs: user shares the doc link, system fetches via Docs API (requires OAuth) or user pastes exported content. For transcripts: direct file upload (JSONL, text).
2. **Highlight**: AI identifies candidate patterns, steps, and reusable elements in the source material. User confirms, rejects, or modifies each highlight.
3. **Co-Author**: Side-by-side view — source conversation on the left, emerging skill definition on the right. AI generates a complete initial skill draft from the highlights. Human edits any field. When human makes a structural edit (add/remove/reorder steps), AI re-proposes affected downstream fields (tools, inputs, outputs, runtime adapters).
4. **Eval Extract**: AI proposes eval cases from the conversation ("it looks like this input produced this output — should that be a test case?"). Human curates — approve, edit, or reject each proposed case.
5. **Target**: User selects target runtime(s). System generates runtime-specific adapters. V1: web + Claude Code adapters. Open claw adapter ships alongside. Cowork adapter deferred.
6. **Test**: User runs the skill against the eval suite in-browser, sees results, iterates. Can re-enter co-author mode to fix issues.
7. **Publish**: Skill is published to the registry. Available via API, MCP, web, CLI.

### API Layer

Thin facade. Every web UI action maps to an API call.

**Collection & Source endpoints:**
- `POST /collections` — create a collection of related sources
- `POST /collections/{id}/sources` — add a source (URL, file, transcript)
- `GET /collections/{id}` — get collection with all sources and extracted content

**Workspace session endpoints (turn-based):**
- `POST /collections/{id}/workspace` — start a workspace session, returns session ID + initial AI analysis of sources
- `GET /workspace/{id}` — get current workspace state (highlights, draft skill, proposed evals)
- `POST /workspace/{id}/highlights/{hid}/accept` — accept a highlight
- `POST /workspace/{id}/highlights/{hid}/reject` — reject a highlight
- `PATCH /workspace/{id}/skill` — human edits the skill draft; response includes AI re-proposals for affected fields
- `POST /workspace/{id}/eval-cases/{cid}/accept` — accept a proposed eval case
- `POST /workspace/{id}/publish` — finalize and publish the skill

**Skill & eval endpoints:**
- `GET /skills` — list published skills
- `POST /skills/{id}/run` — execute a skill in a target runtime
- `GET /skills/{id}/eval` — get eval suite
- `POST /skills/{id}/eval/run` — run eval suite
- `GET /skills/{id}/eval/history` — eval run history

**Note:** The workspace session is stateful on the server (stored in DB, not in-memory). Each API call is a discrete turn — no WebSocket or SSE needed for V1. This keeps the implementation simple while supporting the turn-based interaction model.

### MCP Layer

Exposes the API as MCP tools so AI sessions can interact programmatically:

- `create_collection` — AI session can submit its own transcript for extraction
- `suggest_skill` — given a conversation, suggest what skill could be extracted
- `run_skill` — execute a published skill
- `run_eval` — run a skill's eval suite
- `report_success` — flag a successful session as a potential skill candidate

### Canopy Extension Points

The "extend Canopy" approach requires specific integration with Canopy's existing architecture:

**Reused directly (no changes needed):**
- **Session Analysis** (`src/orchestrator/analysis.py`): Already extracts observations (friction, gaps, patterns, missing capabilities) from Claude Code transcripts. The same extraction logic applies to open claw transcripts.
- **Pattern Detection** (`src/orchestrator/patterns.py`): Cross-session aggregation that finds recurring issues. Reused for cross-source aggregation within a collection.
- **Self-Improvement Tracking** (`src/orchestrator/tracker.py`): Outcome tracking (observation → proposal → success/failure). Reused for eval history and skill improvement tracking.
- **Skill Runner** (`plugins/canopy/skills/`): Can invoke skills across plugin systems via `claude -p`. Reused for runtime execution.

**Extended (new adapters on existing interfaces):**
- **Ingestion Pipeline**: Currently reads Claude Code JSONL transcripts from `~/.claude/projects/`. Needs new adapters for: Slack API threads, Google Docs content, open claw transcripts, raw text/file uploads. Each adapter normalizes to a common "conversation" format that the analysis pipeline already consumes.
- **Proposal System** (`src/orchestrator/proposals.py`): Currently generates `new_tool`, `tool_improvement`, `new_skill` proposals. Extended with a new proposal type: `extracted_skill` — a skill definition derived from a workspace session.
- **Registry** (`registry.yaml`): Currently maps domains → servers → tools. Extended to also register extracted skills with their runtime adapters.

**New (built from scratch):**
- **Collaborative Workspace Engine**: The turn-based session manager that coordinates human edits with AI re-proposals. No existing Canopy equivalent — this is the core new component.
- **Skill Schema & Runtime Adapters**: The runtime-agnostic skill format and per-runtime generation. Canopy has skill definitions but they're Claude Code-specific.
- **API Facade**: REST API layer. Canopy currently runs headless via CLI — no web API exists.
- **Web Frontend**: React/Next.js workspace UI. Canopy has no web frontend today.

**Data model additions:**
- `collections` table: groups of related sources
- `sources` table: individual conversation/transcript records with normalized content
- `workspace_sessions` table: stateful co-authoring sessions
- `skills` table: published runtime-agnostic skill definitions
- `eval_suites` table: living test suites with case history
- `eval_runs` table: execution history with results

### Failure Modes

| Failure | What happens | User sees | Mitigation |
|---------|-------------|-----------|------------|
| Slack URL is private/inaccessible | Ingestion fails, no content fetched | "Can't access this thread. Check permissions or paste the content directly." | Fallback to manual paste |
| Google Doc requires auth | OAuth flow fails or doc is restricted | "Need access to this doc. Share it with [service account] or paste the content." | Fallback to manual paste |
| Source content is too short/empty | AI can't identify patterns | "Not enough content to extract a skill. Try adding more sources to this collection." | Prompt to add more sources |
| AI extraction produces poor quality | Draft skill has wrong steps, missing context | User sees draft and edits it — this is the normal co-authoring flow. The turn-based model means bad AI output is always human-reviewable. | Human-in-the-loop by design |
| Skill run fails against eval | One or more eval cases fail | Eval results with pass/fail per case, diff showing expected vs. actual | User re-enters co-author mode to fix |
| Runtime adapter generation fails | Can't produce valid CC skill or open claw prompt | "Couldn't generate [runtime] adapter. Try a different runtime or simplify the skill." | Fall back to web-only execution |
| LLM API timeout/error during workspace | AI re-proposal fails after human edit | "AI is temporarily unavailable. Your edits are saved — try regenerating." | Workspace state persisted in DB, not lost |
| Cross-source aggregation (future) suggests wrong sources | Unrelated sources grouped together | User manually removes mismatched sources from collection | V1 is manual grouping only |

## V1 Scope Boundary

**What's IN V1 (validation scope):**
- Web UI workspace (the brain — collaborative co-authoring)
- API facade (REST, stateful workspace sessions, turn-based)
- Source ingestion: manual paste/upload only (Slack thread text, transcript files, doc content)
- Runtime adapters: Web (guided workflow in browser) + Claude Code (skill file generation)
- Open claw adapter (similar execution model to CC — ships alongside)
- Eval framework: extract initial cases, run in-browser, human-curated growth
- Auth: V1 is single-tenant, no auth (internal tool). Multi-user auth deferred.
- Database: SQLite for V1 (simple, no infrastructure). Migrate to PostgreSQL when needed.

**What's V1+ (post-validation):**
- Slack API OAuth integration (auto-fetch threads by URL)
- Google Docs API integration (auto-fetch doc content)
- Cowork runtime adapter (pending cowork API spec)
- MCP layer (AI sessions can interact programmatically — ships once workspace is proven)
- Slack bot (proactive "want to make this reusable?" suggestions)
- Automated cross-source suggestion (currently manual grouping)
- Multi-tenant / multi-user auth

**Why this split:** The Assignment says "build the demo first." V1 scope validates the core hypothesis (can non-technical users co-author skills from conversations?) with minimum infrastructure. OAuth flows, MCP tools, and Slack bots don't help validate that hypothesis.

## Canopy Cross-Repo Dependency

The Canopy engine lives in `~/emdash-projects/canopy/` (Python, separate repo). This repo (`canopy-web`) is the new web layer. The relationship:

- `canopy` (Python): engine, session analysis, pattern detection, proposal system, self-improvement tracking
- `canopy-web` (this repo, likely TypeScript/Next.js): API facade, web workspace UI, skill schema, runtime adapters

**Integration approach:** canopy-web calls the Canopy engine via its CLI (`canopy improve`, `canopy patterns`, etc.) or imports Canopy as a Python package. For V1, CLI invocation is simpler — the API server shells out to Canopy commands. For V2, consider a Python microservice or direct library import.

**Runtime adapter execution:** For Claude Code/open claw skills, the API server generates the skill file and the user downloads/copies it. The server does NOT execute `claude -p` on behalf of the user. For the web runtime, skill steps execute server-side via Anthropic API calls (the web UI is the guided workflow — each step calls the API, which calls Claude).

## Open Questions

1. **Slack integration depth** (V1+): Just URL ingestion, or a Slack bot that proactively suggests "want to make this reusable?" when it detects a good thread?
2. **Skill versioning**: How do skills evolve? Git-like versioning? Or overwrite-in-place with eval history?
3. **Canopy convergence**: Does this replace canopy-web as the primary web surface, or is it a new module within canopy?
4. **Eval quality**: How do you judge whether an extracted eval case is actually good? Human curation only, or automated quality signals?
5. **Runtime adapter quality**: AI-generated runtime adapters (e.g., open claw system prompts from abstract steps) are a hard AI problem. V1 should treat adapter generation as AI-assisted + human-reviewed, not fully automatic. Need eval dimension for adapter quality.

## Success Criteria

1. **Neal test**: Neal (CSO) can paste the CRISPR Slack thread + Eva's work into the web UI, co-author a skill with AI assistance, and run that skill against a new topic — within 60 minutes on first use (target: 30 minutes once familiar), without touching a terminal. 4. **Reuse rate**: At least one skill extracted by a non-Jonathan user gets reused by someone else within 30 days.
5. **Canopy integration**: Canopy's existing session analysis can feed sources directly into collections via MCP, without manual intervention.
